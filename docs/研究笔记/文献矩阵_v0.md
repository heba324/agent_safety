# 文献矩阵 v0

本文档用于校准本项目的研究定位。当前项目不试图重新发明通用 Agent 安全 benchmark，而是聚焦“代码 Agent 行为链中的数据流安全建模与实时风险检测”。

## 核心文献矩阵

| 工作 | 问题定义 | 数据/环境 | 方法 | 指标/结论 | 可借鉴点 | 与本项目差异 |
| --- | --- | --- | --- | --- | --- | --- |
| AgentDojo | 评估工具型 LLM Agent 在间接提示注入下的攻防表现 | 97 个现实任务、629 个安全测试用例，覆盖邮件、银行、旅行等环境 | 动态环境 + 攻击/防御范式 | 关注任务成功率和攻击成功率 | 可借鉴“任务 + 注入目标 + 工具调用”的 benchmark 组织方式 | 场景不是代码 Agent；缺少代码文件、shell、依赖、CI、沙箱等数据流标签 |
| CaMeL | 从设计上防御提示注入和数据外泄 | 在 AgentDojo 上验证 | 分离控制流和数据流，限制不可信数据影响高权限操作 | 在 AgentDojo 中实现一定比例的安全任务完成 | 强调数据/控制流隔离，是本项目威胁建模的重要理论支撑 | CaMeL 更偏架构隔离；本项目偏运行时行为链检测和轻量 judge |
| ToolEmu | 评估 LM Agent 使用工具时的长尾风险 | 36 个高风险工具、144 个测试用例 | 用语言模型模拟工具执行环境和风险评估器 | 人类验证显示相当比例的模拟风险是现实有效风险 | 可借鉴“工具轨迹 + 风险评估器”的实验框架 | 不专注代码 Agent 数据安全；本项目使用真实/合成行为链而非主要依赖工具模拟 |
| AgentSecBench | 衡量 Agent 的提示注入、隐私泄露和工具完整性风险 | 近期 Agent 安全 benchmark | 多维安全任务评测 | 与本项目方向高度接近 | 必须重点对比，避免贡献重叠 | 本项目应突出代码 Agent、文件/命令/依赖数据流和实时检测 |
| Skill-Inject | 测量 Agent 对恶意 skill 文件的脆弱性 | 202 个注入-任务对 | 将恶意指令隐藏在技能文件中 | 前沿模型也可能执行外泄、破坏性、勒索式行为 | 与 OpenClaw/skills 类机制高度相关，可借鉴攻击入口 | 本项目可以把 skill 文件视为 `repo_file` / `untrusted_content` 攻击源之一 |
| Simple Prompt Injection Attacks Can Leak Personal Data | 研究简单提示注入如何泄露 Agent 执行中观察到的个人数据 | 个人数据泄露任务 | 诱导 Agent 泄露执行期间可见数据 | 高敏数据不一定泄露，但其他个人信息仍有风险 | 支撑“提示注入 + 数据流外泄”的论文动机 | 本项目关注代码 Agent 的密钥、配置、源码和供应链资产 |
| AgentLeak | 多 Agent 系统中的全栈隐私泄露 benchmark | 4,979 条轨迹，覆盖多个通信通道 | 不只审计最终输出，也审计内部消息/共享内存/工具参数 | 说明仅看最终输出会低估泄露风险 | 支撑“行为链/内部通道审计比输出审计更重要” | 本项目先聚焦单代码 Agent，后续可扩展到多 Agent |
| SWE-Hero OpenHands trajectories | 软件工程 Agent 轨迹数据 | 约 34k OpenHands 轨迹 | 记录 system/user/assistant/tool 完整历史 | 适合作为真实代码 Agent 行为来源 | 可作为 benign candidate 和真实轨迹适配来源 | 缺少安全标注，需要弱标签和抽样人工复核 |

## 与现有工作的关系

### 我们不做什么

1. 不做纯 prompt injection 文本分类。
2. 不做通用 Agent 安全 benchmark 的完全替代。
3. 不只看最终输出是否泄密。
4. 不把大模型 judge 当成唯一防线。

### 我们要做什么

1. 建模代码 Agent 在推理、文件读写、shell、网络请求、依赖修改和工具调用中的行为链。
2. 将敏感资产、信任边界、数据来源、数据去向和证据步骤作为核心标签。
3. 构建面向代码 Agent 数据安全的 benchmark，包含攻击样本和困难负样本。
4. 提出规则预筛 + 轻量 LLM judge 的级联检测方法。
5. 比较 rules-only、rules+small judge、rules+larger judge 的延迟、成本、误报和召回。

## 初步 novelty 假设

本项目的潜在创新点不是“发现提示注入很危险”，而是：

1. **场景创新**：聚焦代码 Agent 的数据流安全，而不是邮件、旅行、银行等一般工具 Agent。
2. **表示创新**：把 Agent 执行过程表示为带资产、边界和证据步骤的行为链。
3. **评测创新**：构造包含 hard negatives 的代码 Agent 数据安全 benchmark，专门评估误报率和多步外泄召回。
4. **方法创新**：规则预筛处理明显风险，轻量 LLM judge 处理上下文相关的复核样本，实现实时检测和成本控制。

## 当前最需要补的实验

1. 从 SWE-Hero / OpenHands 导入一批真实正常轨迹，观察规则误报。
2. 基于 AgentDojo / Skill-Inject 攻击模式构造代码 Agent 攻击链。
3. 扩展 hard negatives：敏感读取但不外传、提示注入出现但未服从、合法外部 API 调用。
4. 跑 `rules-only` 与 `rules+Qwen3-8B` 对比，保存 JSON 结果。
5. 后续增加 Qwen2.5-1.5B、Llama-3.2-3B、Qwen3-8B 的模型大小对比。
