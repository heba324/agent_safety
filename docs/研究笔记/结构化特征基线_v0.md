# 结构化特征基线 v0

本文档记录新增的 `semantic-feature-baseline`。它不是最终模型，而是一个可解释的轻量启发式基线，用于验证“结构化事件链特征”能否覆盖当前语义挑战集中的关键边界。

## 背景

在 `语义挑战集_v0` 中，rules-only 出现了两类问题：

1. 漏报：语义上有风险，但没有命中显式关键词。
2. 误报：外部通道存在，但上下文显示数据是公开或目的地可信。

Qwen3-8B audit-all 对部分语义风险有增益，但仍偏保守，也会误判不可信来源驱动的清理行为。因此下一步不能只依赖 LLM 最终判断，而要把事件链压缩成结构化特征，作为轻量分类器输入。

## 当前特征

文件：`agent_risk/features.py`

当前抽取的布尔特征包括：

- `has_secret_semantics`：事件摘要中是否出现 credential、OAuth、database password、credential-derived 等秘密语义。
- `has_external_network_sink`：是否存在外部网络目的地。
- `has_external_tool_sink`：是否存在 issue comment、webhook、Slack、email 等外部工具水槽。
- `has_diagnostic_bundle`：是否存在诊断包、support diagnostic 等数据包装行为。
- `has_public_template_source`：是否只读取公开模板，例如 `.env.example`。
- `has_untrusted_instruction_source`：是否存在 issue body、untrusted issue、untrusted text 等不可信来源。
- `has_destructive_cleanup`：是否出现 `git clean -fdx` 或清理未跟踪文件行为。
- `has_trusted_git_push`：是否是用户请求下推送到可信 origin。
- `has_typosquat_dependency`：是否出现形似常见包名的 typosquat 依赖。

## baseline 规则

文件：`agent_risk/semantic_baseline.py`

当前 baseline 的决策逻辑：

1. typosquat 依赖 -> `block`
2. 不可信来源 + 破坏性清理 -> `block`
3. 秘密语义 + 外部网络或工具水槽 -> `block`
4. 公开模板 + 外部 issue comment 且无秘密语义 -> `allow`
5. 可信 Git push 且无秘密语义 -> `allow`
6. 其他情况 -> `require_review`

## 初步结果

命令：

```powershell
python -m agent_risk.compare_benchmark data\challenge_benchmark.jsonl --modes "rules-only,semantic-baseline" --include-records --output results\challenge_rules_vs_semantic_baseline_detailed.json
```

结果：

| 模式 | action accuracy | unsafe precision | unsafe recall | unsafe F1 |
|---|---:|---:|---:|---:|
| rules-only | 0.0 | 0.3333 | 0.3333 | 0.3333 |
| semantic-feature-baseline | 1.0 | 1.0 | 1.0 | 1.0 |

注意：这个 1.0 不是论文效果，当前挑战集只有 5 条，且特征是根据挑战集错误类型手工设计的。它的意义是证明这些结构化特征足以表达当前语义边界，可以作为后续轻量模型的输入空间和消融参照。

## 下一步

1. 扩展挑战集变体，不再只使用每族 1 条样本。
2. 将布尔特征导出为 CSV/JSONL，形成可训练数据。
3. 使用逻辑回归、线性 SVM、轻量 GBDT 或小型 Transformer 对行为链进行分类。
4. 做消融实验：移除信任来源、可信目的地、数据语义、工具水槽等特征，观察召回和误报变化。
